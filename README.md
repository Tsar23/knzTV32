## Лабораторна робота №1: Тестування на основі ризику (Risk-Based Testing)

### Мета роботи
Ознайомитись з підходами пріоритезації тестових сценаріїв в залежності від потенційних ризиків для бізнесу. Налаштувати базове середовище автоматизації (Python, pytest, Selenium).

### Аналіз ризиків для системи SauceDemo
При виконанні роботи функціонал сайту був розділений на три категорії ризику:
1. **Critical Risk (Критичний ризик):** Авторизація користувача. Якщо логін не працює, бізнес зупиняється. Тестується першочергово.
2. **Medium Risk (Середній ризик):** Сортування товарів. Важливий функціонал для зручності (юзабіліті), але його поломка не блокує процес покупки.
3. **Low Risk (Низький ризик):** Наявність посилань на соціальні мережі (Twitter) у підвалі сайту. 

### Інструкція з локального запуску тестів (macOS)
1. Активація середовища: `source venv/bin/activate`
2. Запуск усіх тестів: `pytest test_lab1_risks.py -v`
3. Запуск тільки критичних тестів: `pytest -m critical_risk -v`

## Лабораторна робота №2: Сіре тестування (Grey-Box Testing)

### Мета роботи
Застосувати техніки сірого тестування, комбінуючи перевірку користувацького інтерфейсу (UI) з перевіркою внутрішнього стану системи SauceDemo.

### Виконані завдання
1. **Перевірка Cookies:** Написано автоматизований тест `test_session_cookie_created_on_login`, який після авторизації користувача читає збережені браузером cookies (`session-username`), підтверджуючи правильність генерації сесії на рівні даних.
2. **Аналіз Local Storage через ін'єкцію JavaScript:** Реалізовано тест `test_cart_state_in_local_storage` для перевірки кошика. При кліку на кнопку додавання товару, тест виконує JS-скрипт `window.localStorage.getItem('cart-contents')`, щоб переконатися, що система коректно записує ID товару у внутрішню пам'ять браузера.

### Інструкція з запуску тестів другої лабораторної
1. Запуск тестів сірого ящика: `pytest test_lab2_greybox.py -v`

## Лабораторна робота №3: Тестування середовища та конфігурації

### Мета роботи
Провести нефункціональне тестування продукту (сумісність), дослідити адаптивність інтерфейсу в різних середовищах виконання та перевірити роботу системи при зміні конфігурацій користувача.

### Виконані завдання
1. **Емуляція мобільного середовища:** Написано автоматизований тест `test_mobile_environment_ui`, який примусово змінює розмір вікна браузера до роздільної здатності мобільного телефону (390x844). Тест успішно перевіряє появу адаптивного "бургер-меню".
2. **Емуляція десктопного середовища:** Тест `test_desktop_environment_ui` імітує роботу на Full HD моніторі (1920x1080) для перевірки цілісності базового UI.
3. **Тестування конфігурацій доступу:** Тест `test_locked_user_configuration` перевіряє обробку виключень системою, використовуючи альтернативні налаштування даних (обліковий запис `locked_out_user`).

### Інструкція з запуску тестів третьої лабораторної
1. Перевірка середовища та конфігурацій: `pytest test_lab3_env_config.py -v`

## Лабораторна робота №5: Тестові оракули, евристика та техніки тест-дизайну

### Мета роботи
Застосувати динамічні техніки тестового дизайну (Еквівалентне розділення, Таблиці прийняття рішень) для оптимізації кількості перевірок та реалізувати тестові оракули для автоматичного підтвердження результатів.

### Виконані завдання
1. **Техніки тест-дизайну:** Для перевірки логіки авторизації використано метод таблиць прийняття рішень (динамічний аналіз). Замість написання 4 окремих тестів, створено один параметризований тест `test_login_decision_table_and_oracles`, який проганяє 4 класи еквівалентності (валідні дані, заблокований користувач, невалідний пароль, пусті поля).
2. **Тестові оракули та евристика:** Реалізовано інтелектуальний механізм перевірки (оракул), який динамічно змінює очікування залежно від вхідних даних:
   * *Оракул `success_url`:* перевіряє факт зміни URL-адреси.
   * *Оракул `error_message`:* перевіряє появу DOM-елемента з помилкою.
   * *Евристика:* додано додаткову перевірку текстового вмісту помилки, щоб переконатися, що система не просто видає абстрактний збій, а коректно пояснює користувачу причину проблеми.

### Інструкція з запуску тестів п'ятої лабораторної
1. Запуск параметризованих тестів з оракулами: `pytest test_lab5_oracles_heuristics.py -v`
